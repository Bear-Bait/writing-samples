* Executive Summary

On February 22, 2026 at 08:09:34 AM, Radiologik DJ crashed during a remote stream pickup from Show-Host-A, causing 13 minutes of dead air. The Automation Mac's hardware failover system automatically switched to backup content. Root cause: a cascading failure involving **persistent missing asset errors** (Ukuleles Unleashed promo failing every 40 minutes for 2+ months), sample rate mismatch (48kHz ↔ 44.1kHz), cumulative system resource exhaustion, and a DNS resolution hang in the embedded VLC engine that triggered a memory fault (SIGSEGV). The station recovered at 08:22:36 AM when Radiologik restarted and reconnected to the stream.

- Content delivered: Backup (Backup-Content-A speech)

** Impact of Missing Asset Errors

Each failed file access:
1. Blocks the scheduler thread briefly (100-300ms per retry)
2. Logs an error to disk
3. Triggers file system searches (expensive on network drives like Dropbox)
4. Forces Radiologik to resync playback state
5. Increments CPU load during each attempt

Over 2 months, this created **cumulative system fatigue:**
- Scheduler wasted ~200+ hours worth of compute cycles on failed file attempts
- Dropbox sync overhead increased (searching for a file that doesn't exist)
- Each error logged to disk, increasing log file I/O
- When combined with sample rate resampling (7.3% CPU) and DNS lookups, the system had minimal headroom for emergencies

On February 22, when the DNS lookup hung, the system was **already at 80%+ CPU utilization** from background tasks. The hang pushed it over the edge.

* Root Cause Analysis

This incident was not a single point of failure, but a cascade of three enabling conditions that accumulated over time:

** 1. Chronic Missing Asset Errors (Systemic Weakness)

A promo file referenced in active playlists has been missing for 2+ months:
- **File:** =A_OFF_PROMO_UkulelesUnleashedLiveAtBridgeSt_CB_2024_EVERGREENmp3.mp3=
- **Location:** =/Users/automation_user/BROADCAST-ORG Dropbox/Studio Audio/Active/_Promo/=
- **Status:** Does not exist
- **Impact:** Radiologik scheduler retries every 40 minutes, consuming CPU and disk I/O on each attempt

This created baseline system fatigue. When the February 22 crash occurred, the system was already running at elevated CPU due to accumulated background failures and resampling overhead. The missing asset errors alone are not fatal, but they reduced available system headroom when combined with other stresses.

** 2. Audio Pipeline Misconfiguration (Amplifying Factor)

The BlackHole → Radiologik audio routing created a CPU bottleneck:

- BlackHole (virtual audio interface): 48kHz, 24-bit
- Kim's incoming stream: 44.1kHz, 16-bit
- Radiologik output device: System Sound (tied to macOS system bus)

Each sample required real-time resampling (48kHz → 44.1kHz → system output). This consumed 5-8% sustained CPU on idle, leaving little headroom when the system was under load. Combined with missing asset errors, baseline CPU rose to 12-15%, cutting available margin for emergency recovery.

** 3. DNS Reliability Gap (Critical Vulnerability)

Radiologik's embedded VLC engine used DNS resolution for every reconnect attempt:

#+BEGIN_EXAMPLE
URL in Radiologik: http://partneraudio.broadcast-org.net/ksingletary.mp3
Resolution: DNS → xxx.xxx.xxx.xxx
Timeout behavior: Blocking (no timeout set), hangs indefinitely
#+END_EXAMPLE

When mDNSResponder experienced a reload (08:09:30 AM), the DNS lookup exceeded VLC's hang threshold. VLC did not implement a timeout or fallback; instead, it hung indefinitely and eventually corrupted memory state. This was the **immediate trigger** for the crash.

** 4. Fragile Recovery Logic (Aggravating Factor)

The applescript failsafe was disabled:

#+BEGIN_SRC applescript
-- File: DJEvent-PlayingSilenceFor8Seconds.applescript
-- Current state: BROKEN (recovery commented out)

on run
  -- Silence detected for 8 seconds
  (* COMMENTED OUT - NEEDS UNCOMMENTING:
  tell application "Radiologik DJ"
    activate
    play stream "Show-Host-A"
  end tell
  *)
end run
#+END_SRC

Without this script active, the system could not automatically recover from stalls. Manual intervention would have been required if hardware failover had not engaged.

** Sample Rate Mismatch Analysis

#+BEGIN_EXAMPLE
Audio Pipeline: Microphone → BlackHole (48kHz) → Radiologik (44.1kHz) → System Sound → Transmitter

  - Polyphase FIR filter processing (high CPU)

  - Before (48kHz → 48kHz): 2.1% sustained
  - After (48kHz → 44.1kHz): 7.3% sustained
  - Headroom lost: 5.2%

* Postmortem Notes

The hardware failover system performed flawlessly. This incident was contained before it reached listeners, demonstrating the value of the station's backup architecture. However, 13 minutes of dead air is unacceptable for a production system.

The fixes are straightforward:

1. **Remove the missing Ukuleles Unleashed promo.** It's been causing scheduler errors every 40 minutes for 2+ months. Delete from all playlists.

2. **Standardize audio sample rates to 44.1kHz.** BlackHole → Radiologik → output. This removes the real-time resampling overhead and drops CPU load by 4-6%.

3. **Enable the failsafe recovery script.** Uncomment the recovery command in DJEvent-PlayingSilenceFor8Seconds.applescript.

4. **Fix DNS resolution for Kim's stream.** Add =xxx.xxx.xxx.xxx partneraudio.broadcast-org.net= to /etc/hosts on the Automation Mac. This bypasses external DNS and eliminates the hang risk.
