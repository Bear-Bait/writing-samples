#+DATE: 2026-02-27
#+AUTHOR: BROADCAST-ORG Automation Team
#+SEVERITY: CRITICAL
#+STATUS: UNDER_INVESTIGATION
#+COMPUTERS: STATION-A-Catskill-Streams-Mac-mini (Catskill A)

* Executive Summary

LogMeIn audio drivers are hijacking Catskill A's system audio, creating vulnerability. Attempted removal using procedures from STATION-A Troubleshooting Reference Guide, but it failed. Root cause: macOS and LogMeIn updates changed how/where system driver files are stored since those instructions were written. Driver is now located in kernel extension directories protected by System Integrity Protection (SIP). Possible solutions: (1) boot machine into Recovery Mode and disable SIP to manually remove driver, or (2) stop using LogMeIn entirely and test replacement remote access tool that does not install audio drivers (MeshCentral).

* What Was Tried

Followed documented removal steps from troubleshooting guide:

| Attempted Fix | Location Checked | Result | Why Failed |
|---------------|------------------|--------|-----------|
| kextunload | Memory unload | Temporary only - reloads on reboot | Doesn't remove from disk |
| rm /Library/Application Support/LogMeIn/drivers/ | Wrong path | Driver not found there | OS/LogMeIn updates moved file locations |
| rm /System/Library/Extensions/ | Wrong directory | Driver not there | File is in /Library/Extensions, not /System/Library/ |
| sudo rm -rf /Library/Extensions/LogMeInSoundDriver.kext | Correct path, wrong approach | Blocked by SIP | macOS Monterey protects /Library/Extensions with System Integrity Protection |

** Why the Old Fix Doesn't Work

The troubleshooting guide was written before recent OS updates. Newer versions of both macOS and LogMeIn changed:
- Where driver files are stored
- How SIP protects system directories
- Driver architecture itself

Current locations:
- =/Library/Extensions/LogMeInSoundDriver.kext= (active)
- =/Library/StagedExtensions/Library/Extensions/LogMeInSoundDriver.kext= (staged backup)

Both protected by SIP.

* What Could Work (If We Go This Route)

** Option 1: Recovery Mode SIP Bypass (30 min, requires physical access)

1. Power down Catskill A
2. Boot into Recovery Mode (hold power button)
3. Terminal → Utilities
4. Disable SIP: `csrutil disable`
5. Reboot normal
6. Remove driver: `sudo rm -rf /Library/Extensions/LogMeInSoundDriver.kext` + staged copy
7. Re-enable SIP in Recovery Mode
8. Reboot and verify: `kextstat | grep -i logmein` (should return nothing)

Would require on-site staff or physical visit. 20-30 minute job.


* Recommendation: Test MeshCentral 

Instead of fighting with SIP, consider this:

- **MeshCentral** is a graphical remote desktop tool that does NOT install audio drivers
- Would eliminate the "Death Spiral" vulnerability entirely
- Time investment: ~2 hours to test, install, verify on non-critical machine first
- If successful, we migrate all broadcast machines away from LogMeIn

This solves the problem structurally rather than fighting SIP every time OS updates.

* Immediate Action

**Avoid using LogMeIn on Catskill Machine, especially near times it's scheduled to be used for brodcasts.** Use SSH for all remote troubleshooting.
