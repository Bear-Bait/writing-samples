* Root Cause Analysis

** The Death Spiral: How LogMeIn Broke CoreAudio

When a user opens LogMeIn and logs in graphically to a Catskill or Hudson machine, the LogMeInSoundEngine virtual audio driver is instantiated. This driver registers itself with CoreAudio as a priority device.

Upon activation, LogMeInSoundEngine forces CoreAudio to re-evaluate its entire device registry. This triggers the creation of a =CADefaultDeviceAggregate=—a virtual device that attempts to combine all audio sources into a single logical output.

However, the presence of the LogMeIn driver creates a conflict: CoreAudio cannot stabilize on a single aggregate. Instead, it enters a recursive loop:

#+BEGIN_EXAMPLE
18:12:58 HALS_Device::Activate: LogMeInSoundEngine:0 (ID 1303)
18:12:59 HALS_Device::Activate: CADefaultDeviceAggregate-487-41
18:12:59 HALS_Device::_Deactivate: CADefaultDeviceAggregate-487-40
18:14:24 HALS_Device::Activate: CADefaultDeviceAggregate-487-42
18:14:24 HALS_Device::_Deactivate: CADefaultDeviceAggregate-487-41
18:15:27 HALS_Device::_Deactivate: LogMeInSoundEngine:0
18:24:16 HALS_Device::Activate: LogMeInSoundEngine:0 (ID 1366)
18:24:17 HALS_Device::Activate: CADefaultDeviceAggregate-487-43
18:24:17 HALS_Device::_Deactivate: CADefaultDeviceAggregate-487-42
...continuing with -487-44, -487-45... (43 iterations in 42 minutes)
#+END_EXAMPLE

Each aggregate device ID is a new kernel object. Each creation/destruction cycle consumes CPU and I/O resources. After ~43 iterations, the cycle count exceeded the macOS I/O budget, triggering:

#+BEGIN_EXAMPLE
coreaudiod: ERROR: Overload due to HAL client proc exceeding io cycle budget
#+END_EXAMPLE

At this point, CoreAudio daemon crashed or hung. All connections from Radiologik and BUTT to audio devices were severed. The broadcast stream transmitted silence.

** Why Both Machines Failed Identically

- **Catskill A:** Clean node, no custom monitoring stacks
- **Hudson A:** Running Tailscale, MeshCentral, Promtail
- **Result:** Both failed in identical fashion

This confirms the failure is intrinsic to the LogMeInSoundEngine driver and macOS CoreAudio architecture, not environmental.

* Remediation & Prevention

** Protocol: Hard Ban on LogMeIn GUI During Broadcasts

**NEVER use LogMeIn graphical login for remote technical checks during or immediately before a live broadcast.**

Alternatives (safe to use):

- **MeshCentral:** Remote desktop + terminal, does NOT activate audio drivers
- **SSH via Tailscale:** Command-line access, zero audio driver interference
- **Emergency phone support:** Coordinate with on-site staff instead

** Emergency Recovery Procedure

If CoreAudio hangs and audio is lost:

#+BEGIN_SRC bash
# Kill CoreAudio daemon and broadcast apps
sudo killall coreaudiod
killall butt
killall "Radiologik DJ"

# Restart applications
open "/Applications/Radiologik DJ.app"
open "/Applications/butt.app"

# Verify audio is restored
log show --predicate 'process == "coreaudiod"' --last 30s | grep Activate
#+END_SRC

** Long-Term: Eliminate LogMeIn Audio Driver

- Audit all BROADCAST-ORG machines for LogMeIn installation
- Prefer MeshCentral for graphical remote access (no CoreAudio interference)
- If LogMeIn is required, configure it to disable audio driver on broadcast machines
